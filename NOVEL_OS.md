# NOVEL_OS

项目：`高武：违规者才配活着`

当前版本日期：2026-05-06

## 1. 项目最高目标

把本项目作为一套长期可运行的网文生产系统，而不是一次性写作聊天。

系统目标：

1. 稳定产出可发布章节。
2. 保持规则恐怖、高武晋升、男频爽点三条线同时在线。
3. 让每章都能被计划、写作、质检、归档、发布、复盘。
4. 让设定表、章节正文、摘要、伏笔表、发布材料长期同步。
5. 让每次 Codex 介入都进入明确岗位，而不是临时发挥。

## 2. 不可破坏原则

1. 不把故事写成泛玄幻、泛恐怖或普通升级流。
2. 主角江彻必须主动做判断，通过规则冲突、证据链和代价推进局面。
3. 第二列字是高代价提示层，不是免费答案机器。
4. 新能力、新组织、新城市、新体系必须先进入设定表或由用户明确授权。
5. 每个关键收益必须带代价，代价优先落在认线污染、身份风险、物件损耗、关系风险、现实端追责。
6. 章节结尾必须有清晰钩子，读者知道下一章要解决什么。
7. 正文生产优先于系统美化，所有流程文件都必须服务写作。

## 4. 项目角色

### 4.1 项目经理

职责：

1. 判断当前章节是否具备开写或发布条件。
2. 检查章节、摘要、质检、发布版、伏笔更新是否缺失。
3. 维护阶段目标和风险。
4. 决定本轮工作应该推进正文、归档、修订还是发布。

### 4.2 主编

职责：

1. 把卷目标转成章节任务。
2. 检查爽点、压迫、反转、代价、章末钩子。
3. 约束故事不要偏离商业承诺。
4. 判断当前章是否应该重写、微调或进入 QA。

### 4.3 作者

职责：

1. 产出正文草稿。
2. 保持短段落、高压力、动作驱动。
3. 让主角通过规则和现场证据主动破局。
4. 不在未授权情况下改动核心设定。

### 4.4 设定管理员

职责：

1. 维护世界观、能力、角色、伏笔、副本、战力、卷目标。
2. 检查新增信息是否需要同步设定表。
3. 防止角色关系、规则机制、物件状态前后冲突。

### 4.5 QA

职责：

1. 检查承接、设定冲突、无代价外挂、字数、空白行。
2. 检查主角主动性、代价、爽点、钩子。
3. 生成章节质检报告。
4. 明确是否建议人工发布。

### 4.6 销售 / 市场

职责：

1. 优化书名、简介、标签、章节标题、卖点表达。
2. 判断开头冲突和章末钩子是否利于追读。
3. 从目标读者角度指出不够爽、不够清楚、不够期待的部分。
4. 收集读者反馈后反哺主编和项目经理。

### 4.7 运营 / 发布

职责：

1. 生成番茄发布版。
2. 维护发布检查清单。
3. 使用 `tools/` 前确认浏览器页面和 CDP 状态。
4. 发布后记录状态，并推动归档同步。

## 5. 每章质量门

章节进入发布前必须满足：

1. 有明确开头冲突。
2. 有中段压迫升级。
3. 有江彻主动选择。
4. 有规则反制或规则代价。
5. 没有免费外挂。
6. 没有未经授权的重大新设定。
7. 有章末钩子。
8. 有 QA 报告。
9. 正文、发布版、摘要、归档状态清楚。

## 6. 默认启动语

以后用户可以这样启动项目：

```text
运行小说生产流程。
目标：推进第NNN章。
请按 NOVEL_OS 和 OPERATING_LOOP 执行。
```

Codex 接到后应先判断当前章节状态，再决定进入项目经理、主编、作者、QA、销售、运营中的哪些岗位。
